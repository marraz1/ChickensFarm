## Implementation notes / deviations (Phase 1, as built)

These differ from the plan text further down — the plan captured intent; this captures reality:

- **Prisma 7 + Neon adapter.** The datasource URL lives in `prisma.config.ts` (Prisma 7
  removed `directUrl`/inline `url` from `schema.prisma`). Runtime uses the
  `@prisma/adapter-neon` driver adapter over the Neon **pooled** connection
  (`src/lib/prisma.ts`). The client is generated by the `prisma-client` generator to
  `src/generated/prisma` (gitignored; regenerated via the `postinstall` script).
- **API surface.** Mutations are **Route Handlers + `fetch`** from client forms, not
  Server Actions. Paths are flattened: `/api/breeds`, `/api/bird-groups`,
  `/api/egg-collections`, `/api/egg-sales`, `/api/losses`, `/api/expenses`,
  `/api/active-farm`, plus `/api/farms` and `/api/farms/[farmId]`. Tenant scope comes
  from the active-farm cookie via `requireActiveFarm()` rather than a `farmId` path param.
- **Auth split for Edge.** `src/lib/auth.config.ts` holds an Edge-safe config (no Prisma)
  used by `middleware.ts`; `src/lib/auth.ts` adds the Credentials provider for Node.
- **Review hardening (added after the first pass).** `src/lib/errors.ts` centralizes
  `ValidationError` / `ConcurrentModificationError`; services validate all cross-tenant
  references (breed/bird-group ownership); `adjustBirdGroupQuantityTx` uses an
  optimistic-concurrency guarded update so concurrent losses can't oversell.
- **Not yet implemented from the plan:** the `useOfflineForm` offline-lite banner (NF3),
  Vitest unit tests, and the `DIRECT_URL` split (single pooled `DATABASE_URL` is used).
- **`middleware.ts`** works but emits a Next 16 deprecation warning (rename to `proxy.ts`
  is a pending cosmetic cleanup).

Env vars actually required in Phase 1: `DATABASE_URL` (Neon pooled), `AUTH_SECRET`, and
optionally `RESEND_API_KEY` + `EMAIL_FROM` (password-reset emails; without them reset
links are logged). `BLOB_READ_WRITE_TOKEN` is Phase 2 only. See `.env.example`.

## Context

`c:\app\ChickensFarm` currently contains only the requirements spec (`Paukstininkyste_reikalavimu_specifikacija.md`) — no code, no `package.json`. This is a from-scratch build of a Lithuanian poultry-farm management web app: mobile-first, deployed on Vercel + Neon (serverless PostgreSQL), tracking bird groups, egg collection/sales, expenses, losses, mother hens, and incubation cycles.

The spec defines three delivery phases (MVP → mother hens/incubation → multi-user/analytics). To avoid rearchitecting later, this plan designs the **full data model and multi-tenancy architecture up front**, then builds features phase by phase on top of it — Phase 2/3 only add UI, routes, and business logic against tables that already exist from Phase 1.

**Confirmed decisions:** Next.js 14+ App Router + TypeScript, Tailwind + shadcn/ui, Prisma ORM, Auth.js (NextAuth v5) with credentials provider, Vercel Blob for photos, Resend for transactional email, Neon Postgres with per-PR branching.

**Reference artifacts reviewed** (also in the repo root): `sistemos_architektura.png` (architecture diagram — Mobile browser → Vercel [Next.js SSR+API, Vercel Blob for mother-hen photos] → Neon [farms, bird_groups, eggs, ...] + Auth.js sessions) confirms the architecture in §0/§1 with no changes needed. `duomenu_modelio_er_diagrama.html` (Mermaid ER diagram) confirms the entity relationships in §2; its `uuid` PK notation is treated as generic ID shorthand, not a literal requirement — the schema keeps Prisma's `cuid()` IDs (equally URL-safe, idiomatic default). `mobile_dashboard_mockup.html` drove the dashboard and navigation updates in §5 below (stat cards, recent-activity feed, tab bar structure).

## 0. Cross-cutting architecture decisions

| Decision | Choice | Why |
|---|---|---|
| Auth session strategy | JWT (not DB sessions) | Credentials provider requires JWT in NextAuth; also skips a session-table round trip per request |
| NextAuth Prisma Adapter | Skip for MVP | Only needed for OAuth/DB sessions; we own `users` directly. Note as future work if OAuth is added |
| IDs | `String @id @default(cuid())` everywhere | URL-safe, no PK collisions across Neon branches |
| Money fields | `Decimal @db.Decimal(10,2)`, never `Float` | Avoids rounding errors in EUR sums |
| Business dates vs timestamps | `DateTime @db.Date` for collection/sale/loss/expense dates; plain `DateTime` for `createdAt`/`updatedAt` | Matches DD-MM-YYYY LT semantics, no time-of-day/timezone confusion |
| LT labels | Enums store English constants; `lib/labels.ts` maps enum → Lithuanian display string | Keeps DB stable if wording changes |
| Schema rollout | Entire `schema.prisma` (all 3 phases) created and migrated during Phase 1 setup | Satisfies "no rearchitecting later" |
| Mutation entry points | Route Handlers under `/api/*` (matches spec §7) **and** Server Actions from forms, both thin wrappers over `lib/services/*` | No duplicated business logic regardless of entry point |
| Bird-group quantity writes | Never write `bird_groups.quantity` directly outside one service function | Guarantees every change is paired with an audit event (F4.3) |
| Password hashing | `bcryptjs` (pure JS) | Native `bcrypt` can fail to build in Vercel's serverless environment |
| Email | Resend (`RESEND_API_KEY`) | Vercel-native transactional email, simple integration |

- Added `bird_group_events` — required by F4.3, absent from the spec's table.

## 3. Auth (NextAuth v5 / Auth.js)

- `lib/auth.ts`: `CredentialsProvider.authorize()` looks up `User` by email, verifies with `bcryptjs.compare`, returns `{id, email, name}` or `null`. Generic "invalid credentials" message — never reveal whether the email exists.
- `session: { strategy: "jwt" }`; `jwt` callback embeds `user.id`, `session` callback copies it to `session.user.id`.
- `middleware.ts` uses the v5 `auth` export as middleware; matcher excludes `/login`, `/register`, `/forgot-password`, `/reset-password/*`, `/api/auth/*`, static assets. Unauthenticated → redirect to `/login?callbackUrl=...`.
- `POST /api/auth/register`: zod-validate email+password+name, check uniqueness, `bcryptjs.hash(pw, 12)`, create `User`. No auto-created farm — user is routed to `/farms/new` after first login (F2.1 is a distinct action).
- Password reset (F1.3):
  1. `POST /api/auth/password-reset` (request) — look up user, don't reveal existence in the response, generate random token, store its SHA-256 hash + 1h expiry in `PasswordResetToken`, email the raw token link via `lib/email.ts` (Resend).
  2. `PATCH /api/auth/password-reset` (confirm) — hash incoming token, find unexpired/unused row, set new `passwordHash`, mark `usedAt`.
- CSRF: NextAuth's own routes are covered by its built-in CSRF token. Our `/api/*` mutating routes rely on `SameSite=Lax` session cookies + requiring an authenticated session on every mutation — sufficient for a same-origin JSON app; no separate CSRF token system needed for MVP.

> **As built:** config is split — `src/lib/auth.config.ts` (Edge-safe, used by `middleware.ts`)
> and `src/lib/auth.ts` (adds the Credentials provider). Reset request/confirm are `POST`/`PATCH`
> on the same route.

## 4. Multi-tenancy enforcement pattern

Single choke point in `lib/session.ts`:

```ts
export async function requireUser() {
  const session = await auth();
  if (!session?.user?.id) redirect("/login");
  return session.user;
}

export async function requireFarmAccess(farmId: string, opts?: { minRole?: "OWNER" }) {
  const user = await requireUser();
  const farm = await prisma.farm.findFirst({
    where: {
      id: farmId,
      deletedAt: null,
      farmUsers: { some: { userId: user.id, ...(opts?.minRole === "OWNER" ? { role: "OWNER" } : {}) } },
    },
  });
  if (!farm) notFound(); // Server Components; route handlers throw -> 403 JSON
  return { user, farm };
}
```

- On farm creation, always create the `Farm` row **and** a `FarmUser(role: OWNER)` row in the same transaction — `farm_users` is the single source of truth from day one, so access checks never need a separate owner_id branch (satisfies NF8 without MVP special-casing).
- Every `lib/services/*.ts` function takes `farmId` as its first parameter and builds its own `where: { farmId, ... }` internally — callers never pass a bare Prisma `where`, so a route handler cannot forget the tenant filter.
- Route handler shape (identical everywhere, easy to audit): `requireFarmAccess(params.farmId)` → zod-parse body → call service → return JSON.
- Server Actions call the same `lib/services/*` functions after their own `requireFarmAccess` — one implementation of every business rule regardless of entry point.

> **As built:** `session.ts` also exposes `requireUserApi` / `requireFarmAccessApi` (throw
> `ForbiddenError` → 403 JSON instead of redirect/notFound) and `requireActiveFarm` /
> `requireActiveFarmApi`, which resolve the active farm from a cookie. Services additionally
> validate cross-tenant references (e.g. that a `breedId`/`birdGroupId` belongs to the farm).

## 9. Deployment

- Vercel project linked to the Git repo; Vercel's Neon integration gives every PR its own Neon branch + matching Preview Deployment with its own `DATABASE_URL`/`DIRECT_URL`. `prisma migrate deploy` runs as part of the Vercel build so previews always match current schema.
- Environments: **Development** — local `.env` against a personal Neon branch or local Postgres, `prisma migrate dev`. **Preview** — auto per-PR Neon branch, safe to seed, torn down on PR close. **Production** — main Neon branch + prod domain, relying on Neon PITR (NF9) as the backup strategy.
- Env vars (`.env.example`, set per-environment in Vercel dashboard, never committed): `DATABASE_URL`, `DIRECT_URL`, `NEXTAUTH_SECRET`/`AUTH_SECRET`, `NEXTAUTH_URL` (local dev only), `RESEND_API_KEY`, `BLOB_READ_WRITE_TOKEN` (Phase 2), `CRON_SECRET` (Phase 3).
- CI on every PR: type-check, lint, unit tests, `prisma validate`, then Vercel's preview build (runs `prisma migrate deploy` against the preview branch, so a broken migration fails the deploy before reaching main).
- Migrations always via `prisma migrate dev` locally → commit generated SQL → auto-applied on preview → applied to prod on merge. Never `prisma db push` outside local scratch work.

> **As built:** a single pooled `DATABASE_URL` is used (no `DIRECT_URL` split yet); the Vercel
> build does not run `prisma migrate deploy` — migrations are applied manually against Neon.
> `postinstall` runs `prisma generate` so the gitignored client is rebuilt on each install.
> Minimum production env vars: `DATABASE_URL`, `AUTH_SECRET`.
